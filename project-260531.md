이 파일은 **AffiliWrite AI** 프로젝트의 핵심 수집 엔진 및 UI 고도화 이력과 Antigravity 코딩 어시스턴트가 작동할 때 절대 엄수해야 하는 **대원칙 지침**을 보존하여 토큰 소모를 예방하고 일관된 협업 무결성을 유지하기 위한 공식 영구 문서입니다.

## 🚨 [최우선 의무] 소스 코드 수정 및 편집 대원칙

후속 작업에 참여하는 모든 Antigravity 에이전트는 아래의 협업 개발 원칙을 **200% 철저히 준수**하십시오. 이 원칙은 사용자의 명시적 제어 권한을 존중하고 토큰을 아끼기 위한 준엄한 대원칙입니다.

1. **서버 코드 동기화 우선 루틴 (Git Sync-First Workflow)**:
   - 모든 수정 작업을 시작하기 전에, **가장 먼저 서버로부터 최신 코드를 반드시 풀(`git pull origin main`)하여 로컬 코드와 동기화하고 검토**해야 합니다.
   - 로컬 작업 중일 때도 원격 서버의 코드 변경 사항이 있는지 확인하고 항상 서버 기준의 검토 및 수정 단계를 거친 뒤 작업을 진행해야 충돌(Conflict)을 원천 방어할 수 있습니다.
2. **승인 전 소스코드 수정 절대 금지**:
   - 어떠한 버그 핫픽스나 기능 고도화 요청이 들어오더라도, **사전에 코드를 단 1줄이라도 임의로 수정/편집/추가/삭제하지 마십시오.** (로컬 수정 후 '승인 대기' 상태로 대기하는 행위 조차도 절대 불가합니다.)
3. **텍스트 기반의 사전 협의 및 확정**:
   - 이상 현상의 근본 원인과 해결 방안을 **오직 텍스트 응답으로만** 정밀하게 제안하십시오.
   - 제안된 기획 사양에 대해 사용자와 의견을 나누어 내용을 완벽하게 **"확정"**받아야 합니다.
4. **명시적 승인 후 구현 개시**:
   - 사용자의 명시적인 **"승인"**, **"수정해라"**, **"반영해라"**라는 텍스트 사인이 떨어진 후에만 비로소 소스 코드 파일 수정 툴을 최초로 기동하여 작업을 집도하십시오.
5. **배포/푸시 컨펌 절차**:
   - 수정을 마친 후 컴파일 빌드가 통과하면 즉시 푸시하지 말고, 구현된 변경 사항을 사용자에게 팩트 위주로 1차 보고하십시오.
   - 사용자가 검토를 마친 후 **"푸시해라"** 또는 **"배포해라"**라고 명시적 최종 사인을 주었을 때만 비로소 `git push origin main` 명령을 기동하여 실서버 릴리즈를 배포하십시오.

## 🚨 [대원칙] LLM JSON 출력 시 쌍따옴표(") 간섭 및 파싱 에러 방지 규칙

LLM API(Gemini 등)를 사용하여 JSON 구조의 데이터를 동적으로 받아와 파싱할 때, 본문 원고 또는 Frontmatter 텍스트 내부의 따옴표 및 줄바꿈 문자가 JSON 구문 구조를 파괴하여 `Expected ',' or '}'` 파싱 에러를 유발하는 현상을 원천 방지하기 위해 다음 3대 규칙을 영구히 엄수하십시오.

1. **본문 내 영어 쌍따옴표(") 절대 사용 금지 지침 주입**:
   - AI가 작성하는 모든 플랫폼 원고(`naverBlog.content`, `instagram.caption`, `tiktok.audio` 등) 내에서 강조, 인용, 대화, 앵커 텍스트 표시 등을 위해 일반 영어 쌍따옴표(`"`)를 직접 출력하지 않도록 프롬프트에서 강력히 강제해야 합니다.
   - 대신 싱글 따옴표(`'`), 한국식 홑/겹따옴표(`「 」`, `『 』`) 또는 유니코드 스마트 쌍따옴표(`“ ”`)를 사용하도록 규정하십시오. (스마트 쌍따옴표는 유니코드이므로 JSON 구문 기호인 `"`와 분리되어 파싱 에러가 발생하지 않습니다.)

2. **Frontmatter(YAML)의 구조적 객체(Structured JSON) 설계**:
   - YAML Frontmatter나 복잡한 메타데이터처럼 키-값 쌍과 따옴표, 줄바꿈이 빈번하게 얽힌 영역은 프롬프트 상에서 문자열 하나로 퉁쳐서 출력하게 지시(`"frontmatter": "..."`)하지 마십시오.
   - 대신 **정형화된 JSON 객체 구조**(`"frontmatter": { "title": "...", "date": "...", "tags": [] }`)로 설계하여 AI가 네이티브 JSON 스키마를 통해 안전하게 데이터를 구성하도록 만드십시오.

3. **중간 가공 레이어(Adapter Layer) 필수 장착**:
   - API 결과를 파싱(`JSON.parse`)한 후, 화면 렌더링이나 파일 저장 단계로 넘어가기 전에 서비스 레이어(`gemini.js` 등)의 가공 헬퍼 함수(`processMdxFrontmatter` 등)를 거치도록 설계하여, JSON 객체를 화면에 맞는 최종 텍스트 포맷(예: YAML String)으로 안전하게 조립하십시오.
   - 이렇게 하면 AI의 이스케이프 실수를 원천 예방하고 프론트엔드의 동작 안정성을 100% 보장할 수 있습니다.

## 🚨 [대원칙] 100% 무료 플랜 전용 다단계 폴백(Fallback) 체인 운용 규칙

API 서버 부하나 할당량 제한(Rate Limit)으로 인한 작성 오류("This model is currently experiencing high demand...")를 실시간으로 우회하기 위해, 100% 무료 플랜(Free Tier)으로 보장되는 안전 모델군으로 구성된 **다단계 폴백 체인**을 상시 가동하고 엄격히 관리해야 합니다.

1. **사용 가능한 100% 무료 플랜 모델군 명세 (Free-Tier Certified Models)**:
   - **`gemini-3.5-flash`** (1순위): 최신 3.5세대 Flash 모델로 뛰어난 한국어 가독성과 고속 처리를 무상으로 제공합니다.
   - **`gemini-2.5-flash`** (2순위): 검증된 2.5세대 Flash 모델로 오랜 시간 축적된 블로그 글 작성 무결성을 지닌 기본 모델입니다.
   - **`gemini-2.0-flash`** (3순위): 안정적인 2.0세대 Flash 백업 모델입니다.
   - **`gemini-3.1-flash-lite`** (4순위): 3.1세대 최신 Lite 모델로, 대형 서버 혼잡 시간대에도 부하가 거의 없으며 가장 높은 트래픽 우회율을 가집니다.
   - **`gemini-2.5-flash-lite`** (5순위): 2.5세대 Lite 백업 모델입니다.

2. **유료(Pro 계열) 모델의 강력 배제 규칙**:
   - Google AI Studio 정책에 따라 **Pro 계열 모델(`gemini-2.5-pro`, `gemini-3.1-pro-preview` 등)은 무료 플랜에서 제외되어 유료 결제(Pay-as-you-go) 계정에서만 호출 가능**합니다.
   - 무료 API 키 환경에서 Pro 계열 모델을 호출할 시 즉각 호출 실패 에러('Billing not enabled')를 유발하여 폴백 체인 전체가 파괴되므로, **폴백 리스트에 Pro 계열 모델명을 기재하는 행위를 절대 엄금**합니다.

3. **폴백 구현 가이드라인**:
   - `executeWithFallback` 메서드를 통해 지정된 무료 모델 체인을 순차적으로 루프 순회하며 호출해야 합니다.
   - 특정 단계에서 오류 발생 시 해당 모델명과 에러 원인을 `console.warn`에 남겨 투명하게 로그 추적이 가능하게 조치하십시오.
   - 모든 대체 모델 체인이 실패했을 때만 비로소 예외 처리를 화면으로 전달하되, 이때 가장 마지막으로 실패한 모델명 접두사(예: `[gemini-2.5-flash]`)를 에러 메시지에 함께 명기하여 사용자가 문제를 즉시 진단할 수 있도록 구조화하십시오.

## 📂 3대 그룹 트렌드 수집 설정 완전 격리 아키텍처

트렌드 피드 수집 및 가공 엔진은 아래와 같이 **3개 카테고리별로 옵션이 상호 간섭 없이 철저하게 개별 기동**하도록 분할 설계 및 적용되었습니다.

### 1. 설정 구조 계층화 (`trend-rules.json`)
```json
{
  "myInterest": {
    "keywords": ["건강 정보", "경제 재테크", "IT 트렌드"],
    "sources": { "naverBlog": true, "naverNews": true, "naverShopping": false },
    "filtering": { "minCleanScore": 80, "customBlacklist": ["공구", "마켓"], "checkAdRegex": true }
  },
  "naverHotTopic": {
    "keywords": ["다이소 꿀템", "코스트코 추천템", "가전 전자제품"],
    "sources": { "naverBlog": true, "naverNews": false, "naverShopping": false },
    "filtering": { "minCleanScore": 90, "customBlacklist": ["홍보", "체험단"], "checkAdRegex": true }
  },
  "realtimeHotIssue": {
    "keywords": ["AI 인공지능", "신제품 출시"], 
    "sources": { "naverBlog": true, "naverNews": true, "naverShopping": true },
    "filtering": { "minCleanScore": 75, "customBlacklist": ["광고", "협찬"], "checkAdRegex": true }
  },
  "scheduler": {
    "intervalHours": 6
  }
}
```

### 2. 컴포넌트별 믹싱 사양
- **`TrendSettingsPanel.jsx` (트렌드 제어 모달)**:
  - 설정 모달 내부에 `📌 내 관심사`, `🔥 네이버 핫토픽`, `⚡ 실시간 핫이슈` 탭 바를 렌더링합니다.
  - 각 탭 선택 시 해당하는 격리 영역의 키워드 및 채널 범위, 클린 필터 임계치를 상호 간섭 없이 개별 관리 및 병합 저장합니다.
- **`trend-crawler.cjs` (백엔드 수집 엔진)**:
  - 1그룹 '내 관심사' 수집기는 `myInterest` 옵션 기반 기동.
  - 2그룹 '네이버 핫토픽' 수집기는 `naverHotTopic` 옵션 기반 기동 (RSS 공지 덤프를 전면 제거하고, 네이버 검색 API of `sort=sim` 유사도 노출 정렬을 적용해 조회수 최상단 글들을 정교하게 믹싱 수집).
  - 3그룹 '실시간 핫이슈' 수집기는 `realtimeHotIssue` 옵션 기반 기동.

## 🛠️ 260531 1·2·3차 고도화 및 보완 수립 명세

   - 라이트 모드(Morning 테마)의 하얀 배경에서 프론트메타 텍스트 글자색이 연두 형광색(`color: '#a7f3d0'`)으로 고정되어 글자가 아예 실종되던 치명적인 버그를 완전 해결하고, 듀얼 모드 음영 대비가 극대화된 `var(--color-violet)` 변수 색상으로 매핑하여 가독성을 100% 보장하였습니다.
8. **모바일 프리뷰 스크린 세로 스크롤 상시 노출 (`SNSPreviewPane.jsx`)**:
   - 모바일 에뮬레이터 내부 화면의 `overflow-y`를 `scroll`로 강제 고정하고, 얇은 **시안(Cyan) 반투명 전용 세로 스크롤바 디자인**을 명시적으로 주입하여 아래에 풍부한 본문 내용이 더 들어있음을 사용자가 시각적으로 즉각적이고 또렷하게 인지할 수 있도록 마감하였습니다.

## 🧪 테스트 방법 및 예상 결과

### 1. UI 및 연동 테스트 방법 (실서버 운영환경)
1. **모달 진입 및 초기 동기화 검증**:
   - BlogGen 대시보드 우측 상단의 `트렌드 수집 설정` (⚙️) 버튼을 클릭합니다.
   - 모달 내부 상단에 `📌 내 관심사`, `🔥 네이버 핫토픽`, `⚡ 실시간 핫이슈` 탭이 premium 듀얼 테마 스타일에 부합하게 노출되는지 확인합니다.
   - 최초 진입 시 GitHub의 `trend-rules.json` 설정에 따라, 각 탭별로 상이한 키워드와 클린도 점수가 표시되는지 점검합니다.
2. **개별 옵션 격리 및 병합 저장 테스트**:
   - `📌 내 관심사` 탭에서 키워드 하나를 삭제 또는 추가하고, 최소 클린 스코어를 85점으로 변경합니다.
   - `🔥 네이버 핫토픽` 탭으로 전환하여 방금 변경한 내 관심사의 상태가 이 탭에 침범하지 않았는지(원래 키워드와 클린도 90점 유지) 확인합니다.
   - `⚡ 실시간 핫이슈` 탭도 개별 동작함을 검증한 뒤, 우측 하단의 `[트렌드 제어 설정 커밋]` 버튼을 누릅니다.
3. **결과 확인**:
   - 설정 저장이 완료되었다는 확인 메시지가 뜬 뒤, GitHub 저장소의 `trend-rules.json` 파일이 3분할 3-Tier 아키텍처에 부합하는 병합 데이터 구조로 우아하게 커밋 업데이트 되었는지 확인합니다.

### 3. 예상 결과 및 통과 기준 (Expected Results)
* **1그룹 (📌 내 관심사)**:

  * 로그 창에 `[2그룹] 네이버 블로그 실시간 핫토픽 인기글 수집 시작...` 콘솔이 출력됩니다.
